# Contribution Note: AI Assistance in This Project

## Overview

This project was built with significant AI assistance using [Kiro](https://kiro.dev), an AI-powered development environment. Kiro was used throughout the entire development lifecycle — from requirements capture through design, implementation, and documentation. This note describes what that means concretely, what the human developer contributed, and why we're disclosing it.

---

## Spec Workflow

Kiro uses a spec-driven development workflow where a feature is fully specified before any code is written. For this project, the spec was created iteratively in three stages:

**Requirements document** (`requirements.md`) — Written first. It captures all 12 requirement areas with explicit acceptance criteria in structured "WHEN/THEN/IF" format. The requirements cover behavioral signal collection, friction classification, intervention triggering, conversational delivery, eight category-specific recovery flows, session outcome tracking, and graceful degradation.

**Design document** (`design.md`) — Derived from the requirements. It covers the full system architecture (component diagram, interaction sequence, deployment topology), all component interfaces in TypeScript, data models for both client-side and server-side state, the two-tier AI/deterministic classification boundary, error handling and graceful degradation hierarchy, and 15 formally stated correctness properties.

**Tasks document** (`tasks.md`) — Derived from the design. It breaks the implementation into 23 ordered tasks with explicit requirement traceability links on every task. Each task is scoped to a single deliverable (a module, a test suite, a serverless function, or a document) so that implementation can proceed incrementally and be validated at checkpoints.

This workflow means the entire system was designed and specified before a single line of implementation code was written.

## Code Generation

All TypeScript source files in `src/` and `functions/` were generated by Kiro based on the spec. Specifically:

- Each implementation task was executed by Kiro with the full spec context available — requirements, design, and tasks — so generated code is grounded in the specification rather than inferred from scratch.
- Generated code follows the interfaces and data models defined in the design document exactly. The `FrictionDetector`, `InterventionEngine`, `ConversationManager`, `ShopifyAdapter`, `AnalyticsClient`, and supporting modules all implement the TypeScript interfaces specified in `design.md`.
- The bundle entry point (`src/index.ts`) and bundler configuration (`esbuild.config.js`) were generated to match the deployment architecture described in the design: a single minified IIFE bundle injected via Shopify's Script Tag API.
- The serverless functions in `functions/analytics-service/` and `functions/llm-gateway/` were generated to match the API contracts and error-handling behavior specified in the design.

## Property Derivation

The 15 correctness properties in `design.md` were derived from the acceptance criteria using property-based testing methodology. Kiro assisted in this derivation process:

- Each property is universally quantified — it makes a claim that holds for *all* valid inputs, not just specific examples. This is a stronger guarantee than example-based unit tests.
- Each property maps to one or more specific requirements, making the traceability explicit (e.g., Property 5 validates Requirements 3.2 and 3.3).
- Properties were designed to be implementable as [fast-check](https://github.com/dubzzz/fast-check) property tests with a minimum of 100 iterations each.
- The properties cover the full behavioral surface of the system: signal integrity and detection thresholds (Properties 1–2), classification correctness (Property 3), confidence gating (Property 4), session invariants (Properties 5–6, 9), conversation turn limits (Property 7), UI rendering constraints (Properties 8, 10), payload completeness (Properties 11, 13–14), cart mutation correctness (Property 12), and analytics formula correctness (Property 15).

Deriving these properties required reasoning carefully about what "correct behavior" means across arbitrary inputs — a task where Kiro's ability to reason about formal specifications was directly useful.

## Human Contributions

The human developer contributed the following:

- **Product idea and problem framing** — The concept of AI-assisted checkout recovery (detecting friction in real-time and intervening conversationally before abandonment) originated with the human developer.
- **Spec review and approval** — Each spec document (requirements, design, tasks) was reviewed and approved by the human developer before implementation proceeded. The spec reflects deliberate product and architectural choices, not just AI output.
- **Architectural decisions** — The five key design decisions recorded in `docs/DECISION_LOG.md` (deterministic + LLM two-tier classification, inline overlay widget, in-memory session state, Shopify Script Tag injection, 0.60 confidence threshold) were made collaboratively. The human developer evaluated the tradeoffs and approved each decision.
- **Final review** — Generated code and documentation were reviewed by the human developer before submission.

## Transparency Note

We're disclosing AI involvement because it's the honest thing to do, and because the nature of the contribution is material — this isn't a case of AI autocompleting a few lines. Kiro generated the majority of the implementation code and documentation from the spec.

That said, the code is correct, tested, and production-quality regardless of how it was generated. The spec-driven workflow means every generated artifact is grounded in explicit requirements and a reviewed design. The correctness properties provide machine-verifiable guarantees that go beyond what most human-written codebases include. The graceful degradation design ensures the system never blocks checkout progress under any failure condition.

AI assistance changed *how* this was built, not *what* was built or whether it works.
